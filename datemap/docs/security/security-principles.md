# Security Principles

## 핵심 원칙

DearMap은 위치, 사진, 메모처럼 사적인 정보를 다루는 앱입니다. 기능을 빠르게 만드는 것보다 사용자의 기록을 안전하게 다루는 것을 우선합니다.

## MVP 보안 기준

- API 키와 민감 설정은 저장소에 커밋하지 않는다.
- `.env`는 로컬에서만 사용하고 `.env.example`에는 키 이름만 둔다.
- 모바일 앱에는 네이버 Client Secret을 저장하지 않는다.
- 위치 권한은 현재 위치 기능을 사용할 때 요청한다.
- 사진 권한은 사진 첨부 시점에 요청한다.
- 권한을 거부해도 앱이 빈 화면이나 오류 상태로 멈추지 않게 한다.
- 현재 위치는 사용자가 데이트 장소로 저장할 때만 기록한다.
- SQLite에는 필요한 최소 데이터만 저장한다.
- 로그에는 위치 좌표, 메모, 사진 URI, API 키를 출력하지 않는다.

## API 키 관리

- 개발 키와 운영 키를 분리한다.
- 네이버 지도 API 키에는 가능한 앱 패키지, 번들 ID, 도메인 제한을 설정한다.
- 키 유출이 의심되면 즉시 재발급하고 기존 키를 폐기한다.
- Client Secret이 화면 공유, 스크린샷, 로그에 노출되면 재발급한다.
- 서버 호출이 필요한 API는 앱에서 직접 호출하지 않고 백엔드 프록시를 검토한다.

## 로컬 데이터 보호

- 데이트 카드 데이터는 사용자 개인정보로 취급한다.
- 사진 URI와 메모는 디버그 로그에 남기지 않는다.
- 로컬 DB 백업, 내보내기, 공유 기능은 별도 보안 검토 후 추가한다.
- 앱 삭제 시 로컬 데이터가 어떻게 처리되는지 플랫폼별로 확인한다.

## 추후 로그인 기준

- 로그인 기능은 Phase 4에서 추가한다.
- 인증 토큰은 일반 SQLite 테이블에 저장하지 않는다.
- 액세스 토큰과 리프레시 토큰은 보안 저장소 사용을 검토한다.
- 서버 API는 사용자 인증과 권한 검사를 전제로 설계한다.
- 비밀번호 로그인을 직접 구현하기보다 소셜 로그인 또는 검증된 인증 서비스를 우선 검토한다.

## 추후 커플 연동 기준

- 커플 연동은 초대 기반으로 설계한다.
- 각 데이트 카드에는 소유자와 공유 공간 개념이 필요하다.
- 공유된 기록이라도 수정, 삭제 권한은 명확히 나눈다.
- 커플 연결 해제 시 데이터 소유권과 접근 권한 정책을 정의한다.
- 한 사용자가 다른 사용자의 기록에 접근할 수 없도록 서버에서 권한을 검증한다.

## 개발 체크리스트

- 새 API 키를 추가했는가?
- `.env.example`만 커밋하고 실제 `.env`는 제외했는가?
- 위치, 사진, 메모가 로그에 남지 않는가?
- 권한 거부 흐름을 테스트했는가?
- 새 테이블이나 필드가 개인정보를 불필요하게 저장하지 않는가?
- 추후 로그인과 커플 공유에서 소유권 검증이 가능한 구조인가?
